# Access-API

Простой backend-проект на Django + DRF.

## Идея проекта

В проекте есть:

- свой пользователь с логином по `email`
- свой вход и выход через cookie
- своя таблица пользовательских сессий
- своя ролевая система доступа
- простые API для управления ролями и правами
- mock-раздел для примера бизнес-ресурса

Главная идея такая:

- пользователь входит в систему
- для него создаётся запись в таблице `UserSession`
- в cookie кладётся ключ этой сессии
- по cookie проект понимает, кто делает запрос
- дальше доступ проверяется через роли и права

## Как устроен проект

### Пользователи

Пользователь хранится в приложении `apps.users`.

У него:

- `email` вместо `username`
- обычный пароль Django
- `is_active`, `is_staff`, `is_superuser`

Пароль хранится не в открытом виде. Django сам хеширует его через `set_password`.

### Сессии

После логина создаётся запись в таблице `UserSession`.

В ней хранится:

- кому принадлежит сессия
- ключ сессии
- активна ли она
- когда она истекает
- IP и user agent

В cookie сохраняется наш кастомный ключ сессии. На каждом запросе middleware ищет эту сессию и, если всё в порядке, подставляет пользователя в `request.user`.

### Роли и права

Права лежат в приложении `apps.access`.

Там есть 5 основных моделей:

- `Role` — роль
- `Resource` — ресурс
- `Action` — действие
- `UserRole` — какая роль назначена пользователю
- `RolePermission` — что может делать роль с ресурсом

Пример:

- есть ресурс `mock_orders`
- есть действия `read`, `create`, `update`, `delete`
- роли `viewer` можно дать только `read`
- роли `manager` можно дать все 4 действия

Тогда проект будет проверять не просто авторизован ли пользователь, а есть ли у него конкретное право на конкретный ресурс.

## Как работает аутентификация

### Регистрация

`POST /api/users/register/`

Что происходит:

- создаётся пользователь
- пароль хешируется стандартным способом Django
- автоматически создаётся или берётся роль `viewer`
- роль `viewer` назначается новому пользователю

### Логин

`POST /api/users/login/`

Что происходит:

- пользователь отправляет `email` и `password`
- проект проверяет пароль
- создаёт `UserSession`
- ставит cookie
- возвращает данные пользователя

### Логаут

`POST /api/users/logout/`

Что происходит:

- проверяется, что пользователь авторизован
- текущая сессия помечается неактивной
- cookie удаляется

### Текущий пользователь

`GET /api/users/me/`

Возвращает данные текущего пользователя.

### Обновление профиля

`PATCH /api/users/me/`

Обновляет поля текущего пользователя.

### Удаление аккаунта

`DELETE /api/users/me/delete/`

Это мягкое удаление:

- у пользователя ставится `is_active=False`
- все его сессии деактивируются
- cookie удаляется

## Как работает авторизация

Логика сделана просто:

- во view сначала проверяется, что пользователь авторизован
- потом проверяется, есть ли у него нужное право

Например, для mock-заказов могут проверяться такие действия:

- `read`
- `create`
- `update`
- `delete`

Если права нет, возвращается `403`.

Если пользователь не авторизован, возвращается `401`.

## API управления доступом

Есть отдельный набор endpoint-ов для управления ролями и правами:

- список ролей
- список ресурсов
- список действий
- список и создание назначений ролей пользователям
- удаление назначений ролей
- список и создание правил доступа ролей
- удаление правил доступа ролей

Эти endpoint-ы доступны:

- суперпользователю
- или пользователю, у которого есть право `manage` на ресурс `access_rules`

## Django admin

Через admin можно руками управлять всеми основными сущностями:

- пользователями
- сессиями
- ролями
- ресурсами
- действиями
- связями ролей с пользователями
- связями ролей с правами

То есть даже без специальных команд можно вручную собрать нужную схему доступа.

## Mock-раздел

В проекте есть приложение `apps.mock_resources`.

Сейчас там сделан простой пример ресурса `MockOrder`.

Для него есть endpoint-ы:

- `GET /api/mock/orders/`
- `POST /api/mock/orders/`
- `GET /api/mock/orders/<id>/`
- `PATCH /api/mock/orders/<id>/`
- `DELETE /api/mock/orders/<id>/`

Этот раздел нужен как пример того, как писать обычные бизнес-endpoint-ы с нашей системой ролей и прав.

## Структура проекта

- `config/` — настройки проекта и корневые URL
- `apps/users/` — пользователи, сессии, login/logout/me
- `apps/access/` — роли, ресурсы, действия, права доступа
- `apps/mock_resources/` — пример простого бизнес-ресурса

## Настройка базы данных

По умолчанию проект использует SQLite.

Если задать переменные окружения для Postgres, проект будет использовать PostgreSQL:

- `POSTGRES_DB`
- `POSTGRES_USER`
- `POSTGRES_PASSWORD`
- `POSTGRES_HOST`
- `POSTGRES_PORT`

## Запуск проекта

1. Создать и активировать виртуальное окружение.
2. Установить зависимости:

```bash
pip install -r requirements.txt
```

3. Выполнить миграции:

```bash
python manage.py makemigrations
```

```bash
python manage.py migrate
```

4. Создать суперпользователя:

```bash
python manage.py createsuperuser
```

5. Запустить сервер:

```bash
python manage.py runserver
```
